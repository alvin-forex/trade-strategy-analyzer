# Trade Strategy Analyzer — 功能架構思路

> 文檔目的：記錄每個功能 tab 的設計思維、計算邏輯、數據流向、以及「點解咁設計」

---

## 設計哲學

### 核心原則
1. **離線優先** — 唔依賴任何外部 API，所有計算本地完成
2. **單文件架構** — `docs/index.html` 一個檔案包含所有邏輯，方便 GitHub Pages 部署
3. **多維度分析** — 從唔同角度切同一份數據，畀用戶全面理解策略
4. **可操作性** — 每個分析都要有結論同建議，唔係淨係展示數字

### 數據流
```
CSV上傳 → parseCSV() → trades[]
                          ↓
SET上傳 → parseSET() → setParams{}    buildLayerMapping(trades) → layerMap{}
                          ↓                    ↓
              buildPositions(trades, layerMap) → positions[]
                          ↓
              ┌───────────┼───────────┐
              ↓           ↓           ↓
        calcEntryScore  calcStats  calcSymbolStats
        calcStrategyScore  ↓        calcLayerStats
              ↓         calcSessionStats
        calcFinalScore  calcDirectionStats
                          ↓
              ┌───────┬───┴───┬───────────┐
              ↓       ↓       ↓           ↓
         基礎分析  市場背景  深度分析    存檔
```

## Tab 設計思路

### 1. 📈 收益曲線（Equity Curve）
**點解**：用戶第一眼最想睇到嘅嘢 — 「呢個策略總共賺/蝕幾多」
- SVG 折線圖展示累積盈虧走勢
- 正負區域著色（綠色盈利、紅色虧損）
- 關鍵統計卡片放在圖下方，一眼捕捉核心指標
- **設計決定**：用 SVG 而非 Canvas — 輕量、可縮放、唔使額外庫

### 2. 🎯 質量評分（Quality Scoring）
**點解**：唔係淨係睇盈虧，要評估「入場質素」同「策略管理能力」

**兩層評分設計思維**：
- **Entry Score（入場）**：評估「時機好唔好」
  - 方向(35%) — 趨勢對唔對（用 Max Pips 唔係 Net Pips，因為 Max Pips 代表最大潛力）
  - 時機(35%) — 入場後有幾多浮盈 vs 浮虧
  - 初始回撤(30%) — 入場即刻捱打幾耐
- **Strategy Score（策略）**：評估「管理好唔好」
  - 回歸率(30%) — 深層加倉後能否自救（馬丁策略最關鍵嘅指標）
  - 出場效率(25%) — 賺到幾多 vs 本來可以賺幾多
  - 風控(20%)、盈利品質(15%)、成本(10%)
- **Final = Entry×0.4 + Strategy×0.6** — 策略管理比入場時機更重要（0.6 權重）

**馬丁修正（L4+ timing ×1.2）**：深層加倉本身就代表逆勢，時機分數應該有 bonus

### 3. 💱 貨幣對（Symbol Analysis）
**點解**：用戶想知「邊個貨幣對最賺錢」
- 按 PF 排序 — PF 比「總盈利」更能反映策略穩定性
- 點擊展開詳細 — 避免一開始太長
- 顏色編碼（綠=好、紅=差）

### 4. 📚 層數分析（Layer Analysis）
**點解**：馬丁格爾策略嘅核心風險就在層數深度

**分組邏輯**：
- L1 only — 最安全，理想情況應佔大多數
- L1-L2 — 可接受
- L1-L3 — 開始有壓力
- L4+ — 危險區域

**健康指標**：
- L1-only 佔比 ≥ 50% = 健康（大部分交易唔使加倉）
- L4+ 佔比 ≤ 20% = 可控
- L4+ 回歸率 ≥ 70% = 就算加倉都能自救

### 5. 🕐 時段分析（Session Analysis）
**點解**：外匯市場 24 小時運作，唔同時段波幅同特性完全唔同

**時段定義（HKT）**：
- 亞洲盤 00-08 — 波幅細，適合 range 策略
- 歐洲盤 14-22 — 波幅大，流動性高
- 美洲盤 21-05 — 跨午夜，跟歐洲重疊時段最活躍
- **重疊處理**：亞洲盤優先匹配 0-4（因為美洲盤定義跨午夜，但亞洲開盤更重要）

**24h 逐小時棒型圖**：精細到每個小時，搵出最佳交易時段

### 6. ⏱️ 持倉時間（Holding Time）
**點解**：持倉時間反映策略類型（日內 vs 波段 vs 長線）

**5 段分組**：
- <30m — 剝頭皮型
- 30m-1h — 短線
- 1-4h — 日內
- 4-12h — 波段
- 12h+ — 長線

**用分鐘而唔係小時**：因為大部分外匯 EA 持倉偏短，小時精度唔夠

### 7. 🌐 市場語境（Market Context）
**點解**：策略好唔好，要結合當時嘅市場環境先知

**方案 A 設計**：
- 用本地 MT4 D1 數據（唔使 API key，完全離線）
- 顯示同期價格走勢 + 買賣方向標記
- 自動判定「順勢/逆勢」— 幫用戶理解策略喺咩環境下表現好

### 8. 🔄 市況×策略（Market Regime × Strategy）
**點解**：方案 A 只係背景，方案 C 要「分類 + 對比」

**市況分類算法**：
```
每日判定流程：
1. ATR > 1.5× 50日均值？ → 高波動 🟠
2. |SMA20-SMA50|/中位數 < 0.15% 且 RSI 40-60？ → 震盪 🟡
3. SMA20 > SMA50 且 RSI > 50？ → 上升趨勢 🟢
4. SMA20 < SMA50 且 RSI < 50？ → 下降趨勢 🔴
5. 以上皆非 → 混合信號，按 RSI 方向判定
```

**優先級設計**：
- 高波動優先判定（因為高波動環境風險最大）
- 震盪第二優先（明確嘅中性狀態）
- 趨勢判定最後（因為 SMA 差距計算需要 threshold）

**雙 CSV 對比**：
- 同一份 CSV = 同一個信號頁面嘅交易結果
- 兩份唔同 CSV = 同一個 EA 用唔同 SET 跑同一個信號
- 按同一市況切片對比 → 消除市場因素，純粹比較 SET 好壞

### 9. 🏆 Copy 推薦（Copy Trade Matrix）
**點解**：用戶最終想知「應唔應該 Copy 呢個策略」

**矩陣設計**：
- 行 = 貨幣對，列 = 層數深度
- 每格 = 等級(A/B/C/D) + 關鍵指標
- **L4+ 最高 B 級** — 深層加倉無論如何都唔應該推薦 Copy

**等級標準**：
- A：WR≥60% + PF≥1.5 — 可以放心 Copy
- B：WR≥50% + PF≥1.2 — 可以，但要注意
- C：WR≥40% + PF≥1.0 — 勉強，需要調整
- D：其他 — 不建議

### 10. 📏 波幅分析（Volatility Analysis）
**點解**：策略表現好唔好，同市場波幅有直接關係

**兩個分析角度**：
1. **層數波幅** — CSV 自帶數據（Max Pips + |Max Loss Pips|）
2. **ATR 交叉分析** — 本地 MT4 數據（將 ATR 轉為 pips）

**Pip 乘數設計**：
- 5 位小數對（AUDCAD 等）×10000
- JPY 對 ×100
- XAU/XAG ×0.1（因為 ATR 單位係美元，要轉為「美元/0.1盎司」）

**三分組（p33/p67）**：
- 低波幅 — ATR < P33
- 中波幅 — P33 ≤ ATR ≤ P67
- 高波幅 — ATR > P67

### 11. 🎲 馬丁風險（Martin Risk）
**點解**：馬丁格爾策略嘅最大風險就係「爆倉」

**風險等級**：
- LOW：最大層數 ≤ 3
- MEDIUM：最大層數 4-5
- HIGH：最大層數 6-7
- CRITICAL：最大層數 ≥ 8（接近 slInLevel=9）

**回歸率**：L4+ 倉位最終盈利嘅比例 — 反映策略嘅「自救能力」

### 12. 📐 TP/SL 分析
**點解**：止損止盈設定直接影響策略表現

**優化建議公式**：
- SL = Max Loss Pips P90 × 1.2（覆蓋 90% 回撤 + 20% 緩衝）
- TP = Max Pips P75（75% 倉位曾到達此盈利）
- TSL = 差距中位數（Max Pips - Net Pips 的中位數 = Trailing Stop 距離）

**DollarMode 分析**：從 SET 參數提取出場機制設定

### 13. 💰 Copy 模擬（Copy Simulation）
**點解**：理論分析唔夠，要模擬「如果我咁 Copy 會點」

**兩種模式**：
- **跟勝（Copy on Profit）**：只跟 L1-L2，設獨立止損 → 保守型
- **跟虧（Copy on Lose）**：跟所有層數但設上限 → 願意承受風險但控制最大虧損

**止損自動填入**：從 TP/SL 分析嘅建議值自動填入

### 14. 📁 歷史記錄（Archive）
**點解**：用戶會重複分析同一個信號頁面，需要追蹤表現變化

**設計**：
- localStorage 存檔（5MB 上限）
- CSV hash 做重複偵測
- 智能標籤從檔名提取（signal_14581 → Signal ID: 14581）
- 趨勢迷你圖（勝率/PF/盈虧）— 睇出策略表現隨時間嘅變化

## 技術決策記錄

### 為什麼用單文件 HTML？
- GitHub Pages 只需要一個 `index.html`
- 零構建步驟
- 用戶可以離線打開
- 壞處：文件越來越大（4365 行），但暫時可接受

### 為什麼用本地 MT4 數據而唔是 Yahoo Finance？
- Yahoo Finance CORS 限制，需要代理
- Finnhub 免費版外匯燭線 403
- 本地 .hst 數據：完整歷史（7.5 年）、無 API 限制、即時可用
- 壞處：需要手動 export_hst.py 轉換

### 為什麼用動態層數檢測？
- 唔同 EA、唔同貨幣對嘅手數模式唔同
- 固定比率（0.02=L1, 0.04=L2...）唔適用於所有情況
- 動態排序唯一手數 → 映射層數 更準確

### 為什麼用 floor-to-minute 而唔是 60s tolerance？
- 後端 Python 用 floor-to-minute
- 60s tolerance 會產生唔一致的倉位數（測試中差咗 9 個）
- 前後端對齊比「更準確」更重要

## 已知限制

1. **無回測引擎** — 唔可以模擬「如果用唔同 SET 會點」，只能分析已經跑出嚟嘅結果
2. **D1 精度** — 市況分類用日線，對於 M5/H1 EA 可能唔夠精細
3. **單文件限制** — 4365 行已經開始難維護，未來可能需要拆分
4. **無多用戶** — localStorage 只支持單用戶，唔支持跨設備同步
5. **5MB 存檔上限** — 大量歷史分析可能爆限
